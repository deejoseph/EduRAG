# EduRAG 智能写作助手 — 项目开发进度

> 基于 RAG（检索增强生成）的本地化 K12 教育辅助系统，专注高考语文写作训练。
> 技术栈：Python + ChromaDB + BGE Embedding + Ollama (Qwen3:8B) + Flask + React + Vite + TypeScript + Ant Design + Recharts

---

## 项目结构

```
EduRAG/
├── core/                    # 核心引擎
│   ├── db_manager.py        # ChromaDB 数据库管理
│   ├── embedding.py         # BGE 中文向量嵌入
│   ├── llm_client.py        # Ollama LLM 客户端
│   ├── rag_pipeline.py      # RAG 查询流水线
│   ├── reranker.py          # BGE 重排序模型
│   └── retriever.py         # 向量检索 + 元数据过滤

│   └── routes/
│       ├── __init__.py      # 通用过滤条件构建
│       ├── search.py        # 知识库检索 4 接口（含 RAG 热门主题统计）
│       ├── writing.py       # 写作训练 4 接口
│       ├── practice.py      # 强化训练 7 接口
│       ├── portfolio.py     # 作品集管理 10 接口
│       ├── hot_topics.py    # 命题热点与题库 8 接口

│   └── chinese/

│       ├── writing.py       # 写作训练器封装
│       └── prompts/         # 4 套 Prompt 模板
├── scripts/                 # 数据导入工具
```

| **多目录搜索API** | `api/routes/search.py` | 智能文件查找，支持从多个目录递归搜索（all/ > pdfs/ > docs/ > ...） |

**核心功能：**
- 文档扁平化：794个文档（652 PDF + 142 DOCX）合并到 `data/docs/all/`
- 智能文件查找：优先从 all/ 目录查找，向后兼容其他目录

- MIME类型识别：根据文件扩展名设置正确的Content-Type

**API端点：**
- `GET /search/files/<path:subdir>/<filename>` - 文件访问服务（智能多目录查找）
- `GET /search/files/list?type=<pdf|docx|txt|md>&recursive=<true|false>` - 文件列表

**技术实现：**
```python
# 文件搜索优先级（从高到低）
1. data/docs/all/           # 扁平化目录（最高优先级）
2. data/docs/<subdir>/      # 请求的子目录
3. data/docs/pdfs/          # PDF默认目录
4. data/docs/docs/          # DOCX默认目录
5. data/docs/docs_flat/     # DOCX扁平目录
6. data/docs/texts/         # TXT默认目录
7. data/docs/markdowns/     # MD默认目录
8. data/upload_tmp/         # 上传临时目录
```

### JSON 解析容错系统 & 一键收藏恢复

#### 问题背景
用户搜索"责任担当"等关键词时，LLM 返回的 JSON 频繁出现格式错误，导致解析失败。同时用户发现命题热点模块的"一键收藏"按钮丢失。

#### 已完成工作

**1. 智能 JSON 解析器 `_safe_parse_json_array`** ✅
- **核心问题**：LLM 在 JSON 字符串值内部使用中文引号（\u201c\u201d），简单替换会导致字符串提前结束
- **解决方案**：逐字符状态机扫描器
  - 在 JSON 结构区域正常处理引号
  - 在字符串值内部遇到中文引号时自动转义为 `\"`
  - 遇到不合理的 ASCII 引号（后面不是 `:,}]`）也自动转义
  - 处理未转义的换行符 `\n` / `\r`
- **多层降级策略**：
  1. 首次尝试标准解析
  2. 失败后尝试截断恢复（提取到最后完整的 `}`）
  3. 再失败则替换所有中文标点后重试
  4. 最后组合降级+截断

**3. 相关度评分说明** ✅
系统中存在两种相关度评分机制：
- **向量相似度评分**（RAG 检索）：基于 BGE Embedding 的余弦相似度，范围 0-1，由 ChromaDB 返回
- **LLM 主观评分**（命题热点）：LLM 生成的 `relevance_score` 字段，范围 1-10，表示话题与高考命题的关联程度

#### 技术要点

**状态机 JSON 解析器核心逻辑**：
```python
# 关键判断：字符串内的引号是结构结束符还是内容装饰
rest = raw_json[i + 1:].lstrip(' \t\n\r')
if rest and rest[0] in ':,}]':
    # 后面是结构字符 -> 字符串结束引号
    result.append(ch)
    in_string = False
else:
    # 后面是普通字符 -> 内容中的装饰引号，转义
    result.append('\\"')
```

**第二轮修复（中文引号作为 JSON 结构引号）**：
- **根因**：LLM 有时用 `\u201c`/`\u201d`（中文双引号）代替 `"` 作为 JSON 字符串的开头/结束引号
- **修复**：状态机在**字符串外部**遇到 `\u201c` 时，将其转换为 `"` 并进入字符串模式；在字符串内部遇到 `\u201d` 时，使用与 ASCII 引号相同的 look-ahead 策略判断是内容还是结构结束符
- **截断恢复**：将 `rfind('}')` 替换为正向扫描的 `_find_last_complete_brace()`，正确处理字符串内花括号和嵌套深度，准确找到最后一个完整对象
