### 2. TTS生成器模块 (`podcast/tts_generator.py`)

**核心功能：**
- ✅ **封装LongCat-AudioDiT调用**：支持命令行参数传递
- ✅ **长文本自动分段**：智能分句 + 段间停顿控制
- ✅ **音频拼接**：使用soundfile库合并多个音频段
- ✅ **临时文件管理**：自动清理中间文件
- ✅ **错误处理**：超时检测、错误日志记录

**技术参数：**
- 默认NFE步数：18（质量与速度的平衡）
- 引导方式：APG（优于CFG）
- 引导强度：3.5（防漏字、抗破音）
- 段间停顿：200ms（自然呼吸感）
- 每段最大字数：40字（避免过长导致断句不准）

### 3. 后端API端点 (`api/routes/writing.py`)

**新增接口：**
```python
POST /writing/podcast-tts
```

**请求格式（multipart/form-data）：**
```typescript
{
  text: string,              // 播客文案
  ref_audio: File,           // 参考音频（3-8秒）
  prompt_text: string,       // 参考音频对应的文本

}
```

**响应格式：**
```json
{
  "success": true,
  "audio_url": "/podcast-audio/podcast_1719034338000.wav",
  "duration_sec": 45.2,
  "message": "语音生成成功"
}
```

**静态文件服务：**
```python
GET /podcast-audio/<filename>
```
提供生成的音频文件访问

## 🚨 用户反馈的关键问题（2026-06-21）

根据实际使用体验，发现以下需要优先解决的问题：

### 1. 长文本断句错误严重 ⚠️ **高优先级**

**问题描述：**
- 每次生成的目标文本不能太长，太长了就很容易出现断句错误
- LongCat-AudioDiT 在处理超过50字的段落时，容易出现语义断裂、语调不自然

**解决方案：**
```
播客文案生成后 → 拆分成小段文字（每段≤30字）→ 逐段喂给LongCat
```

**实施计划：**
1. 修改 `text_preprocessor.py` 中的 `smart_split()` 方法
   - 将默认 `max_chars` 从 40 降低到 25-30
   - 增强语义完整性检查（确保主谓宾结构完整）
   
2. 改进 TTS 生成流程
   ```python
   # 原方案：一次性生成长文本
   generate_speech(long_text)  # ❌ 容易断句错误
   
   # 新方案：分段生成 + 人工验证 + 拼接
   segments = smart_split(long_text, max_chars=25)
   for seg in segments:
       audio_file = generate_segment(seg)  # ✅ 短文本更准确
       save_to_validation_dir(audio_file)
   
   # 用户验证后，再拼接
   validated_files = get_validated_audios()
   final_audio = concatenate(validated_files)
   ```

3. 创建音频验证目录结构
   ```
   data/podcast_audio/
   ├── sessions/
   │   ├── session_20260621_173000/
   │   │   ├── seg_001.wav      # 第1段音频
   │   │   ├── seg_002.wav      # 第2段音频
   │   │   ├── seg_003.wav      # 第3段音频
   │   │   ├── validation.json  # 验证状态
   │   │   └── final.wav        # 拼接后的最终音频
   ```

### 2. 英文缩略词读法错误 ⚠️ **高优先级**

**问题描述：**
- AI 会被读成"爱"（应该是字母 A-I）
- CEO 可能被读成"西奥"而不是逐个字母
- GDP、NBA 等常见缩写也有类似问题

**当前问题：**
```python
# text_preprocessor.py 中的词典
"AI": "人工智能",  # ❌ 这样转换不对！应该读字母
```

**正确方案：**
```python
# 应该转换为拼音或字母读音
"AI": "A I",           # ✅ 逐个字母读
"CEO": "C E O",        # ✅ 逐个字母读
"NBA": "N B A",        # ✅ 逐个字母读
"GDP": "G D P",        # ✅ 逐个字母读

# 只有真正需要翻译的才转换
"NASA": "美国航空航天局",  # ✅ 这个是机构名，应该翻译
"WHO": "世界卫生组织",    # ✅ 这个是机构名，应该翻译
```

**实施计划：**
1. 重新分类英文缩写词典
   - **类别A：逐个字母读**（AI、CEO、GDP、NBA、FBI、CIA...）
   - **类别B：按单词读**（NASA、UNESCO、FIFA...）
   - **类别C：翻译成中文**（WHO、UN、EU...）

2. 扩展词典至 200+ 词条
   ```python
   LETTER_BY_LETTER = {
       "AI", "CEO", "CFO", "CTO", "COO", "CIO",
       "GDP", "CPI", "PPI", "GNP",
       "NBA", "FIFA", "NFL", "MLB", "NHL",
       "FBI", "CIA", "NSA", "DEA",
       "USB", "CPU", "GPU", "RAM", "ROM", "SSD",
       "HTML", "CSS", "HTTP", "FTP", "TCP", "IP",
       # ... 更多
   }
   ```

3. 添加正则表达式识别未收录的缩写
   ```python
   # 识别大写字母组合（2-6个字母）
   import re
   pattern = r'\b[A-Z]{2,6}\b'
   matches = re.findall(pattern, text)
   # 对未收录的缩写，默认按字母逐个读
   ```

### 4. 分段生成 + 验证 + 拼接工作流 ⭐ **核心改进**

**新工作流程：**
```
1. 用户上传参考音频 + 输入参考文本
   ↓
2. 输入播客文案（可以很长）
   ↓
3. 系统智能分句（每段≤25字）
   ↓
4. 逐段生成音频，保存到 session 目录
   ↓
5. 展示所有音频段，用户可以：
   - ✅ 试听每一段
   - 🔄 重新生成某一段（调整参数）
   - 🎤 手动录制某一段（如果TTS效果不好）
   - ❌ 删除不满意的段落
   ↓
6. 用户确认所有段落后，点击【拼接音频】
   ↓
7. 系统拼接所有段落，生成最终音频
   ↓
8. 提供下载和分享功能
```

**技术实现：**
1. 创建 Session 管理器 (`podcast/session_manager.py`)
   ```python
   class TTSSession:
       def __init__(self, session_id):
           self.session_id = session_id
           self.segments = []  # 音频段列表
           self.validation_status = {}  # 每段的验证状态
       
       def add_segment(self, audio_path, text):
           """添加音频段"""
       
       def validate_segment(self, segment_id, status):
           """标记段落验证状态（approved/rejected/recorded）"""
       
       def concatenate_all(self):
           """拼接所有已验证的段落"""
   ```

2. 前端展示界面
   ```
   ┌─────────────────────────────────────────┐
   │ 音频段列表（共5段）                      │
   ├─────────────────────────────────────────┤
   │ 1. [▶️ 试听] 你好，世界。     [✅ 通过] │
   │ 2. [▶️ 试听] 这是第二段。     [🔄 重生成]│
   │ 3. [▶️ 试听] 这段不太好。     [🎤 录音] │
   │ 4. [▶️ 试听] 第四段内容。     [⏳ 待验证]│
   │ 5. [▶️ 试听] 最后一段。       [⏳ 待验证]│
   ```

### 5. API调用稳定性评估 🔍 **待验证**

**问题：** 通过 API 调用 LongCat 是否稳定？

**测试方案：**
1. **压力测试**：连续生成 10 段音频，观察成功率
2. **并发测试**：同时发起 3 个生成请求，检查是否会冲突
3. **长时间运行**：持续使用 1 小时，观察内存泄漏和性能下降

**可能的风险：**
- LongCat 模型加载耗时较长（首次约 30-60 秒）
- GPU 显存占用高，可能影响并发
- subprocess 调用可能存在超时或崩溃风险

**备选方案：**
如果 API 调用不稳定，考虑：
1. **方案A**：保持 LongCat 作为独立服务运行（Gradio UI）
   - EduRAG 通过 HTTP 请求调用 Gradio API
   - 优点：稳定，模型常驻内存
   - 缺点：需要维护两个服务

2. **方案B**：在 EduRAG 中直接调用（当前方案）
   - 每次生成时加载模型
   - 优点：架构简单，统一管理
   - 缺点：首次加载慢，资源占用高

**决策标准：**
- 如果成功率 > 95%，采用方案B（直接调用）
- 如果成功率 < 95%，采用方案A（独立服务）
